# External Roadmap

This is the thin public execution roadmap for HELPER.

It is intentionally outcome-based rather than a full internal task board.

## Planning Rule

This document is meant to help external readers understand what the product is trying to achieve next.

It is not:

- a promise of delivery dates
- a source-complete internal execution plan
- a guarantee that parity will be achieved
- a substitute for private planning or private diligence

## Current Starting Point

The public repository already shows:

- a curated trust and diligence surface
- a runnable runtime-review slice
- product-shape evidence through screenshots, artifacts, and docs
- honest public status language that still stops short of a parity claim

## Near-Term Outcomes

### Product And Runtime

- expand the public runtime-review proof slice in ways that remain public-safe
- improve operator-facing flows for research, planning, generation, and runtime review
- strengthen runtime observability and review ergonomics

### Trust And Evaluation

- keep public status language claim-safe and auditable
- move from status shorthand toward a clearer public methodology for future parity-related claims
- improve the blinded-review corpus and evidence governance

### Showcase And Diligence

- keep the public repo synchronized with the current public-safe pack
- improve the quality of the diligence reading path
- preserve a clean boundary between showcase material and private core

## Mid-Term Outcomes

### Product And Runtime

- broaden the set of reviewable operator workflows represented in public-safe surfaces
- improve validation around generated artifacts and runtime outputs
- make the public proof path deeper without exposing sensitive private-core logic

### Trust And Evaluation

- run a fixed-window blinded evaluation under stable rules
- produce a claim-safe public summary if the evaluation window completes
- make it easier for outside readers to understand what has and has not been proven

### Showcase And Diligence

- support more serious partner, reviewer, and acquirer conversations
- keep ownership, proof, and risk notes aligned with the live public repo

## Longer-Horizon Outcomes

- qualify for a stronger public evidence position only if the published thresholds are actually met
- widen the set of public-safe technical proof surfaces without collapsing the private boundary
- make the public repository a higher-signal diligence entrypoint for product, trust, and execution review

## Current Blockers

- the public repository is intentionally not the full private core
- the blinded reviewer corpus is not yet authoritative
- the counted parity window has not started
- proof and claim surfaces must stay stricter than marketing pressure

## What This Roadmap Avoids

- fixed launch dates that are not yet defensible
- promises about hosted SaaS availability
- promises of human-level parity before the evidence exists
- disclosure of private implementation detail that should remain outside the public repo
